I continue with a description of the proposed language.


THE LANGUAGE

Name of Language: iStar-RL

Overview and Motivation: iStar-RL is a goal-oriented language for modeling reinforcement learning agents. When designing reinforcement learning agents, we need to somehow describe the available actions, outcomes, and quality or preferability of outcomes. iStar-RL is a high-level conceptual modeling language that allows us to define exactly that. As in most conceptual modeling languages, there are two categories of concepts: entities, which are distinct types of individuals to be modeled, and relationships, which represent an association or interaction between two or more entities.

Language Concepts:
The entities of iStar-RL are:
        - Actors
        - Goals
        - Tasks
        - Effects (satisfying)
        - Effects (non-satisfying)
        - Effects (indirect)
        - Qualities
        - Pre-Effects
        - Pre-Qualities

The relationships supported by iStar-RL are:
        - wants-to
        - is-AND-refined-to
        - is-OR-refined-to
        - affects (deterministically)
        - affects (non-deterministically)
        - contributes-to (positively)
        - contributes-to (negatively)
        - contributes-to (with no special positive or negative quality)
        - precedes

Detailed Description of Concepts:

Actor (entity): An actor is an active, autonomous entity that holds beliefs and aims at achieving goals. It can be, for example, a human or, more likely in our case, an artificial intelligence system. Examples of actors include:
        - Student (in an education context),
        - Nurse (in a health care context),
        - Participant (in a scientific study like this one),
        - Flight controller (in an aviation context).

Goal (entity): Goals are states of affairs that an actor wants to achieve. Here are some examples:
        - Have Course Requirements Fulfilled,
        - Have Book Acquired,
        - Have Medical Test Completed,
        - Have Medical Supplies Replenished,
        - Have Experimental Tasks Accomplished.

Task (entity): A task is an attempt at an action by an actor, usually with the purpose of achieving some goal. Here are some examples of tasks:
        - Enroll Course,
        - Complete Homework Assignment,
        - Write Exam,
        - Acquire Blood Sample,
        - Order Supplies Online,
        - Purchase Supplies in Person from the Store.


wants-to (relationship): We use the relationship "wants-to" to signify that an actor wants a goal achieved or a task performed. For example:
        - "Student" wants-to "Have Course Requirements Fulfilled"
        - "Student" wants-to "Have Book Acquired"

is-AND-refined-to (relationship): The relationship "is-AND-refined-to" shows how a goal can be decomposed into multiple goals or tasks. All of the child goals and/or tasks must be satisfied (or successfully performed) for the parent goal to be fulfilled. For example, to achieve the goal "Course Requirements Fulfilled", a student needs to enroll in the course, complete the homework assignment, and write the exam. All of these need to be performed, so it is an AND-refinement:
        - "Course Requirements Fulfilled" is-AND-refined-to "Enroll Course", "Complete Homework Assignment", "Write Exam".

is-OR-refined-to (relationship): The relationship "is-OR-refined-to" shows how a goal can be decomposed into alternative subgoals or tasks. Fulfillment or successful performance of just one child is sufficient for the fulfillment of the parent goal. For example, to fulfill the goal "Have Supplies Replenished", an inventory manager may "Order Supplies Online" or "Purchase Supplies in Person from the Store". Either one suffices:
        - "Have Supplies Replenished" is-OR-refined-to "Order Supplies Online", "Purchase Supplies in Person from Store"

Quality (entity): A quality is an attribute for which an actor desires some level of achievement. For example:
        - "Develop New Skills"
        - "Reduce Costs"
        - "Expedite Delivery"

As opposed to goals, qualities do not have a clear-cut definition, i.e., a way to unambiguously decide when they are satisfied or not. Rather, qualities are assumed to be satisfied to some degree based on what tasks actors perform. We represent this through contribution relationships. When we say that an origin element "contributes-to" a target element, we denote that performance or satisfaction of the origin affects satisfaction of the target. Targets of contribution links are always qualities. Origins of contribution links can be goals, tasks, effects, or other qualities.

We specifically have the following types of contributes-to relationships:

contributes-to (positively) (relationship): We use "contributes-to (positively)" if we know that satisfaction or performance of the origin supports the satisfaction of the destination. The following example expresses that purchasing online (versus going to the local store) is typically cheaper:
        - "Order Supplies Online" contributes-to (positively) "Reduce Costs"
        - "Eligible for Scholarship" contributes-to (positively) "Maximize Financial Support"

contributes-to (negatively) (relationship): We use "contributes-to (negatively)" if we know that satisfaction or performance of the origin hurts the satisfaction of the destination. For example, ordering online implies delivery delays when compared with shopping at the local store:
        - "Order Supplies Online" contributes-to (negatively) "Expedite Delivery"

contributes-to (relationship): We use plain "contributes-to" if satisfaction or performance of the origin has an unspecified, unknown, or complex effect on the satisfaction of the destination:
        - "Purchase Supplies Online" contributes-to "Product Quality"

To describe states of affairs that emerge from the performance of a task, we use effects. For example, the task "Write Exam" has possible effects: "Exam Passed with High Grade", "Exam Passed with OK Grade", "Exam Failed", and "Participation Was Not Possible". These effects form the effect group of the task. We represent the relationship between tasks and effects through the "affects (non-deterministically)" relationship.

affects (non-deterministic) (relationship): We use an "affects (non-deterministic)" relationship to relate a task to a group of effects that may come about once the task is performed:
        - "Write Exam" affects (non-deterministically) "Exam Passed with High Grade", "Exam Passed with OK Grade", "Exam Failed", "Participation Was Not Possible".

When a task brings about only one effect, or if an effect has a unique other indirect effect (more below), the deterministic version of the affects relationship is used:

affects (deterministic) (relationship): We use an "affects (deterministic)" relationship to relate a task or effect to a single (other) effect:
        - "Exam Passed with High Grade" affects (deterministically) "Eligible for Scholarship" (here both entities connected are effects).
        - "Write Exam" affects (deterministically) "Eligible for Student Loan"

Not all effects have the same status in a group of effects with respect to a task. If a student's task is to "Write Exam", the first three of the outcomes signify accomplishment of the task. The exam was written even if it was failed. So the first three effects are what we call task-satisfying effects: when they occur, the parent task can be considered accomplished. However, the fourth effect, "Participation Was Not Possible", is non-satisfying of the task. For whatever logistical reason, the student was not able to participate (e.g., they did not have their name on the examinee list), so the exam was not actually written.

To summarize, we have the following types of effects:

Effect (satisfying) (entity): An effect (satisfying) describes an outcome of a task that signifies successful performance of the task. For example:
        - "Exam Passed with OK Grade"
        - "Exam Passed with High Grade"

Effect (non-satisfying) (entity): An effect (non-satisfying) describes an outcome of a task that signifies failure of the task. For example:
        - "Participation Was Not Possible"

We use a separate entity to represent effects that are caused by other effects:

Effect (indirect) (entity): An effect (indirect) describes an outcome that is the indirect result of some other effect. For example:
        - "Eligible for Scholarship" (as an indirect effect of another effect "Exam Passed with High Grade")

Note that we refer to effects as entities to denote that they are unary. Effects represent features of states of the world. From that viewpoint they are indistinguishable from goals (also features of states of the world), except that goals are marked as desired by an actor.

We use precedence constraints to describe restrictions on the order in which tasks are to be performed in time. When a goal, task, or effect is the origin of a "precedes" relationship with a task, that task cannot be performed unless the origin goal is satisfied, the origin task has been performed, or the origin effect has occurred.

precedes (relationship): A goal, task, or effect needs to be satisfied, performed, or true before a target task can be performed. For example:
        - "Write Exam" precedes "Dispute Grade"
        - "Exam Passed with High Grade" precedes "Take More Advanced Course"

The final entities to consider are pre-effects and pre-qualities. Let us see an example to understand pre-effects. Consider a robotic agent that performs a simple task of pressing a light switch. Once pressed, the light switch will turn the lights of a room on if they were off, or turn them off if they were on. So what would the outcome of the task "Press Switch" be? We use a pre-effect element to describe the state of the environment before a task was performed. In our example, we use a pre-effect to describe the status of the lights as it was before the current press of the switch button. This pre-effect, together with the effect of the current press action, will determine the indirect effect of whether the lights will be on or off. Hence, the robot presses the button, and the pressing will most likely succeed.

The same idea can be applied to qualities. Suppose that my quality is to "Improve Understanding of iStar-RL", the language we discuss here. Every time I read this text it may add to or subtract from my understanding of the language. In other words, my new understanding of iStar-RL will be based on my previous understanding of iStar-RL plus or minus the effect that reading the text had on my understanding. We represent this by adding "Improve Understanding of iStar-RL" as a pre-quality element in our model.

Pre-Effect (entity): The previous state of an effect.

Pre-Quality (entity): The previous state of a quality.

Pre-effects and pre-qualities are connected with other elements using the same relationships that effects and qualities use, except that pre-effects and pre-qualities serve as origins in those relationships. In a model, the same description may appear as both an effect and a pre-effect, or as both a quality and a pre-quality. However, a pre-quality cannot be the destination of a contributes-to relationship, and a pre-effect cannot be the destination of an affects relationship.

The case is about an engine manufacturer.

Powerful Engines Inc. is an Engine Manufacturer located in North America. They design and assemble small combustion engines for a small customer list of power tool manufacturers, with many of whom they have long-term contracts. Every time one of those customers places an order, the Engine Manufacturer needs to Have that Order Fulfilled by a strict deadline. To do so, they need to do two things. First, they need to Acquire the Parts necessary for assembling the engine from a variety of suppliers. The parts may be Acquired in Time, may be Acquired Late, or they may be Never Acquired before the absolute deadline. In the latter case, the Order is Cancelled because the company cannot honor the time commitment to the client. According to the contract, a Penalty is Issued for two consecutive cancellations.
